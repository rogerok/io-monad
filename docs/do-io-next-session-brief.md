## Текущая учебная цель

Ученик готовится реализовать `doIo` без вложенной рекурсии. Перед этим ему нужно уверенно понимать:

- что IO-программа в этом проекте является обычным объектом-описанием действий;
- что построение IO-программы не выполняет эффекты;
- что эффекты выполняет только `runIO`;
- зачем у узлов вроде `readLine` и `fetch` поле `next` является функцией;
- почему `doIo` тоже должно вернуть IO-значение, а не выполнить программу сразу;
- почему генератор внутри `doIo` должен получать результат эффекта обратно через `gen.next(value)`;
- почему один и тот же `myProgram` должен быть пригоден для нескольких запусков через разные `world`.

## Где остановились

Ученик начал понимать такую модель:

- через `doIo` IO-монада превращается в программу, записанную генератором;
- `yield` отдаёт наружу очередной IO-узел;
- `runIO` выполняет этот IO-узел;
- результат выполнения, например строка из `readLine`, должен вернуться обратно в генератор;
- после возврата результата генератор продолжает строить следующую часть программы.

Последнее трудное место:

> doIo должен вернуть такое IO, чтобы runIO потом выполнил readLine и вернул результат обратно в генератор.

Это почти правильная интуиция. В следующем разговоре стоит закрепить, что:

- `doIo` не запускает `runIO`;
- `doIo` строит IO-значение, которое при интерпретации будет по одному шагу двигать генератор;
- `readLine.next` и `gen.next(value)` похожи по роли: оба являются продолжением вычисления после получения значения;
- `next` в типе IO и `next` у генератора не одно и то же API, но они выражают похожую идею продолжения.

## Материал ментора

Тема: Часть 8. Do-notation через генераторы.

Ключевые пункты:

- `doIO` принимает фабрику генератора `() => IOGen<A>`, а не сам генератор, потому что генераторы мутабельные и одноразовые.
- Наивная версия: `yield io`, затем внешний код вызывает `gen.next(value)`, а в TypeScript приходится писать `as string`.
- `doIO` возвращает `IO<A>`, то есть программу-значение, а не `() => IO<A>`.
- Адаптер `_` прячет каст и позволяет писать `yield* _(readLine)`.
- Вариант `Symbol.iterator` позволяет писать `yield* readLine`, но это более сложный этап.
- Для текущей ближайшей задачи не нужно начинать с `Symbol.iterator`; сначала важно понять базовую механику `doIo`.

## Рекомендуемый план следующего объяснения

1. Начать не с кода `doIo`, а с уже существующих узлов `IO`: `pure`, `writeLine`, `readLine`, `fetch`, `suspend`.
2. Показать, как `runIO` идёт по объектам в цикле и почему `current = current.next(result)` является продолжением программы.
3. Отдельно проговорить разницу:
   - `readLine.next(value)` строит следующий IO-узел;
   - `gen.next(value)` продолжает генератор после `yield`.
4. Связать эти две идеи: `doIo` должен сделать мост между IO-продолжениями и генераторным продолжением.
5. После этого обсуждать контракт `doIo`, но не писать готовую реализацию, пока ученик явно не попросит.
