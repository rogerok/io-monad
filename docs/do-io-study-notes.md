# Конспект: как работает IO-интерпретатор и зачем нужен doIo

## 1. IO-программа это описание, а не выполнение

В этом проекте IO-программа представлена объектами:

- `pure(value)` означает: вычисление уже завершено и содержит `value`;
- `writeLine(text)` означает: потом нужно вывести строку;
- `readLine` означает: потом нужно прочитать строку;
- `fetchUrl(url)` означает: потом нужно получить тело ответа;
- `sleep(ms)` означает: потом нужно подождать;
- `suspend(thunk)` означает: потом нужно лениво построить следующий IO-кусок.

Когда ты пишешь `writeLine("hi")`, в консоль ещё ничего не выводится. Создаётся объект-инструкция. Побочный эффект появится только тогда, когда этот объект попадёт в `runIO`.

Функциональная идея здесь такая: мы отделяем описание программы от её исполнения. Это похоже на список команд, который можно передать интерпретатору.

## 2. Что делает runIO

`runIO` получает два аргумента:

- `io`: описание программы;
- `world`: конкретный внешний мир, умеющий читать, писать, делать fetch и sleep.

Дальше `runIO` держит переменную `current` и в цикле смотрит на `current.tag`.

Если `current.tag === "writeLine"`, интерпретатор вызывает `world.writeLine(current.text)`, а потом переходит к `current.next`.

Если `current.tag === "readLine"`, интерпретатор вызывает `world.readLine()`, получает строку, а потом вызывает `current.next(res)`.

Главное место:

```ts
const res = await world.readLine();
current = current.next(res);
```

`readLine` не знает заранее, что делать дальше, потому что дальнейшая программа зависит от введённой строки. Поэтому `next` у `readLine` является функцией: ей нужно передать результат чтения.

## 3. Зачем нужен next

`next` у IO-узлов хранит продолжение программы.

Для `writeLine` результат эффекта неинтересен, поэтому `next` может быть уже готовым IO-объектом.

Для `readLine` результат важен, поэтому `next` это функция:

```ts
next: (line: string) => IO<A>
```

Смысл: "когда строка будет прочитана, передай её сюда, и я скажу, какая инструкция следующая".

В функциональном программировании это называется continuation: продолжение вычисления.

## 4. Почему bind превращается в лесенку

До генераторов программа строится через вложенные продолжения:

```ts
bind(readLine, (name) =>
  bind(readLine, (age) =>
    writeLine(`${name}: ${age}`)
  )
)
```

Это честно описывает зависимость: сначала получить `name`, потом с ним строить следующую часть, потом получить `age`.

Проблема не в идее, а в форме записи: код уезжает вправо и становится тяжело читать.

## 5. Что меняет генератор

Генератор позволяет записать те же зависимости линейно:

```ts
const program = doIo(function* () {
  yield writeLine("Name?");
  const name = yield readLine;
  yield writeLine(`Hi, ${name}`);
});
```

`yield readLine` значит: "останови генератор и отдай наружу IO-инструкцию `readLine`".

Потом внешний код должен выполнить эту инструкцию, получить строку и вернуть строку обратно в генератор через `gen.next(value)`.

То есть генератор не заменяет `runIO`. Генератор помогает удобно построить IO-программу, где каждый следующий шаг зависит от результата предыдущего эффекта.

## 6. Два разных next

Есть два `next`, и их легко спутать.

`next` внутри IO-узла:

```ts
readLine.next("Alice")
```

Это продолжение IO-программы после прочитанной строки.

`next` у генератора:

```ts
gen.next("Alice")
```

Это продолжение JavaScript-генератора после `yield`.

Они не одно и то же, но идея похожая: "вот результат прошлого шага, продолжай".

`doIo` как раз должен связать эти две механики.

## 7. Что должен вернуть doIo

`doIo` должен вернуть `IO<A>`, то есть обычное IO-значение.

Он не должен вернуть функцию `() => IO<A>`.
Он не должен сам вызвать `runIO`.
Он не должен сразу выполнить все эффекты.

Правильная ментальная модель:

```ts
const myProgram = doIo(function* () {
  // линейное описание программы
});

await runIO(myProgram, world);
```

`myProgram` это программа-значение. Её можно положить в переменную, передать в функцию, запустить на тестовом `world`, потом запустить на другом `world`.

## 8. Почему doIo принимает фабрику генератора

Генератор одноразовый. Если создать генератор один раз и сохранить его внутри программы, второй запуск может начать уже с исчерпанного состояния.

Поэтому `doIo` принимает не готовый генератор, а функцию:

```ts
() => IOGen<A>
```

Каждый запуск интерпретатора должен получить свежий генератор.

## 9. Что важно понять перед реализацией без вложенной рекурсии

Для версии без вложенной рекурсии ключевая цель не "сразу придумать хитрый цикл", а сохранить контракт:

- программа строится чисто;
- выполнение остаётся в `runIO`;
- результат каждого эффекта возвращается в генератор;
- длинная линейная программа не должна ломаться из-за глубокой вложенности;
- одно и то же IO-значение должно корректно запускаться больше одного раза.

Если эти свойства сохраняются, реализация движется в правильную сторону.
