Как `doIo` связывает генератор, `YieldWrap` и `runIO`

Цель этого файла: разложить по шагам, почему `yield* readLine` отдаёт наружу `YieldWrap(readLine)`, как результат эффекта возвращается обратно в генератор, и зачем `suspend` нужен для отложенного следующего шага.

Это объяснение специально написано как карта понимания. Оно показывает форму движения данных и типы значений, но не является отдельным готовым ответом для копирования в домашку.

1. Базовая модель: IO это объект-инструкция

Построение программы

Когда код вызывает writeLine("hi"), консоль ещё не трогается. Создаётся объект, который описывает будущий эффект. 

writeLine("hi")
// примерно:
{
  tag: "writeLine",
  text: "hi",
  next: pure(undefined)
}

Исполнение программы

Эффекты исполняет только runIO. Он смотрит на tag, вызывает метод из world, потом переходит к следующему IO. 

case "readLine": {
  const text = await world.readLine();
  current = current.next(text);
  break;
}
IO Описание действия: что потом выполнить.
runIO Интерпретатор: выполняет IO-объекты в цикле.
world Реальный внешний мир: readLine, writeLine, fetch, sleep.

2. Что меняет `yield*` и `Symbol.iterator`

В наивной версии генератор пишет yield readLine, и наружу выходит сам readLine. В версии с Symbol.iterator пользователь пишет yield* readLine. Тогда JavaScript вызывает readLine[Symbol.iterator](), а твой mkIO отдаёт наружу обёртку YieldWrap(readLine).

Наивный стиль 
const name = yield readLine;

Наружу из генератора выходит голый IO.

result.value // IO<string>
Текущий стиль 
const name = yield* readLine;

Наружу из генератора выходит обёртка.

result.value       // YieldWrap<IO<string>>
result.value.value // IO<string>

Почему `yield*` возвращает типизированное значение

yield* делегирует управление вложенному iterable. В твоём случае iterable живёт прямо на IO-объекте. Внутренний генератор в mkIO сначала отдаёт YieldWrap(this), а потом возвращает то значение, которое ему передадут обратно через next(value). 

Object.defineProperty(io, Symbol.iterator, {
  enumerable: false,
  value: function* () {
    return (yield new YieldWrap(this)) as A;
  },
});

3. Интерактивный пример: как значение возвращается в генератор

Нажимай "Следующий шаг". Симулятор показывает не настоящий стек JavaScript, а учебную модель: что находится у генератора, что отдаётся в runIO, и какой результат возвращается обратно.

Программа
const program = doIo(function* () {
  yield* writeLine("Name?");
  const name = yield* readLine;
  yield* writeLine(`Hi, ${name}`);
});

4. Зачем здесь `suspend`

suspend нужен не для того, чтобы выполнить эффект. Он нужен, чтобы отложить построение следующего куска IO до момента, когда runIO сам до него дойдёт.

Коротко: suspend(thunk) это IO-узел с отложенной функцией. Когда программа только строится, thunk не вызывается. Он вызывается только в ветке case "suspend" внутри runIO.
Опасная ментальная модель

"Я сейчас внутри doIo пройду генератор до конца обычным циклом". 

while (!done) {
  const result = gen.next(???);
}

Проблема: для следующего gen.next(value) нужно значение прошлого эффекта. Например, строка из readLine ещё не существует.

Рабочая ментальная модель

"Я делаю один шаг генератора, возвращаю IO для runIO, а следующий шаг откладываю". 

yielded IO
  затем bind получает result
  затем suspend откладывает следующий gen.next(result)

В итоге runIO остаётся главным циклом, который разворачивает программу.

Две разные причины для suspend

В твоём варианте suspend полезен в двух местах, и их не стоит смешивать.

  • Внешний suspend вокруг всей do-программы даёт свежий генератор на каждый запуск.
  • Внутренний suspend после каждого эффекта откладывает следующий gen.next(result).

Почему не продолжить сразу

Callback у bind должен вернуть следующий IO. Если он сразу вызывает следующий шаг генератора, вся цепочка начинает раскрываться прямо в момент построения продолжения. Если он возвращает suspend, следующий шаг остаётся обычной IO-инструкцией, которую позже обработает цикл runIO.

Короткая формула

без suspend:
  следующий шаг может строиться прямо сейчас

с suspend:
  следующий шаг будет построен, когда runIO дойдёт до suspend

Поэтому suspend помогает не строить глубокую вложенную лестницу заранее. Он превращает "сразу продолжить" в "вернуть инструкцию: продолжить потом". 

runIO видит suspend:
  current = current.thunk()

и только здесь происходит следующий шаг генератора

5. Глава: suspend как отложенное продолжение

Если убрать детали типов, suspend отвечает на один простой вопрос: "можно ли не делать следующий шаг прямо сейчас, а записать его как инструкцию для интерпретатора?" В нашей IO-модели ответ: да, следующий шаг можно завернуть в thunk.

Что такое thunk

Thunk — это функция, которая держит вычисление внутри себя. Она не вычисляет результат в момент создания. Она ждёт, пока кто-то её вызовет. 

const eager = writeLine("hi");
// IO-объект строится сейчас

const lazy = () => writeLine("hi");
// IO-объект построится только при lazy()

Внешне это маленькая разница. По управлению программой — огромная: теперь у нас появляется ручка "когда именно раскрыть следующий шаг".

Что делает suspend

suspend превращает thunk в обычный IO-узел. Для runIO это такая же инструкция, как readLine или writeLine, только смысл у неё другой: "раскрой меня и получи следующий IO". 

{
  tag: "suspend",
  thunk: () => IO<A>
}

Важная граница: suspend не выполняет внешний эффект. Он только откладывает создание следующего описания программы.

Запомнить: readLine откладывает чтение строки, а suspend откладывает построение следующего IO-узла. Это разные уровни ленивости.

Почему это важно именно для doIo

Генератор нельзя прокрутить до конца заранее. После строки const name = yield* readLine следующий шаг зависит от настоящего результата readLine. До исполнения world.readLine() имени нет. 

const name = yield* readLine;
yield* writeLine(`Hi, ${name}`);

Поэтому doIo должен работать порциями: получить один yielded IO, вернуть его интерпретатору, дождаться результата, а следующий gen.next(result) оформить как будущий шаг.

Продолжить прямо сейчас 
callback(result) {
  return step(result);
}

Ментально это говорит: "я уже сейчас раскрываю следующий кусок программы". На коротких сценариях это может казаться нормальным, но управление снова уходит в цепочку вызовов функций.

Вернуть следующий шаг как IO 
callback(result) {
  return suspend(() => step(result));
}

Ментально это говорит: "следующий шаг существует, но пусть его раскроет runIO". Так главный цикл остаётся в одном месте.

Trampoline-идея простыми словами

Trampoline — это приём: функция не вызывает следующий шаг глубже и глубже, а возвращает "следующий шаг" наружу. Потом внешний цикл сам вызывает эти шаги один за другим.

Почему suspend не memoization?

Memoization означает "посчитать один раз и запомнить". Наш suspend этого не обещает. Он просто хранит функцию. Если runIO раскроет этот узел снова, thunk может вызваться снова.

Почему suspend не async?

Асинхронность живёт в world.readLine(), world.fetch() и других эффектах. suspend сам по себе не делает Promise. Он управляет порядком построения IO-описания.

Синие узлы важны для понимания suspend: генератор двигается на один шаг, а следующий шаг откладывается как IO-инструкция.

7. Частые ошибки

Ошибка 1: перепутать `yield` и `yield*`

Если doIo ждёт YieldWrap<IO>, тесты и программы должны использовать yield*. Обычный yield отдаёт наружу голый IO.

Ошибка 2: передать следующий IO обратно в генератор

Когда done: false, yielded значение — это инструкция, которую ещё надо выполнить. В gen.next(...) надо возвращать результат уже выполненного эффекта, а не саму инструкцию.

Ошибка 3: пройти генератор до конца сразу

Нельзя заранее дойти до следующего yield, если предыдущее значение ещё не получено из world. Например, имени пользователя ещё нет до выполнения readLine.

Ошибка 4: создать генератор слишком рано

Генераторы одноразовые. Поэтому doIo принимает фабрику генератора, а свежий генератор должен появляться на запуск программы, а не быть общим для всех запусков.
