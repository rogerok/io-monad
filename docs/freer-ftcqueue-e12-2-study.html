E12.2: Freer + FTCQueue

Учебная глава для реализации части 12.2: заменить одно продолжение в Freer на очередь продолжений, чтобы left-associated цепочка bind строилась за линейное время и не сжигала JS-стек.

Context. В текущем проекте Freer уже отделил ядро монады от инструкций: pure, impure, cont, bind, freerRun. E12.2 не про новые эффекты. Это замена внутреннего представления continuation chain.

Overview

Наивный Freer хранит в каждом impure одно продолжение: 

type Freer<I, A> =
  | { tag: "pure"; value: A }
  | { tag: "impure"; op: I; cont: (resp: unknown) => Freer<I, A> };

Когда вызывается bind(m, f), он должен приклеить f после всего уже накопленного вычисления. В наивной версии это делается рекурсией: если m — impure, новый cont вызывает старый cont, а затем снова bind.

FTCQueue меняет одну вещь: вместо одного cont внутри impure лежит очередь продолжений. Новый bind больше не обходит дерево, а добавляет функцию в хвост очереди.

До

Impure(op, cont)

bind перестраивает continuation рекурсивно.

После

Impure(op, queue)

bind добавляет continuation в очередь.

Takeaway. Цель E12.2 — не “ускорить функцию”, а изменить структуру данных, в которой хранится будущая работа.

Почему появляется O(N²)

Оператор bind похож на append, который проходит левый аргумент. Если строить цепочку слева, каждый следующий bind снова проходит всё накопленное слева. 

let prog = pure(0);
for (let i = 0; i < N; i++) {
  prog = bind(prog, (x) => pure(x + 1));
}

На шаге k структура уже содержит примерно k звеньев. Стоимость построения: 1 + 2 + ... + N = O(N²). Для маленьких программ это шум. Для traverse, генераторов и больших циклов это становится архитектурным дефектом.

Takeaway. Проблема не в том, что bind “медленный”. Проблема в том, что он выполняет structural traversal при каждом добавлении продолжения.

Новая модель Freer

В Haskell freer-simple использует форму: 

data Eff effs a
  = Val a
  | forall b. E (Union effs b) (Arrs effs b a)

type Arr effs a b = a -> Eff effs b
type Arrs effs a b = FTCQueue (Eff effs) a b

Для TypeScript без HKT и GADT можно держать ту же инженерную идею, но ослабить типы внутри очереди до unknown. Важно сохранить внешний API: FreerIO<A>, freerPure, smart constructors, freerRun, freerDo.

Целевая форма типов

type Cont<I, A, B> = (a: A) => Freer<I, B>;

type FTCQueue<I, A, B> =
  | { tag: "leaf"; f: Cont<I, A, B> }
  | { tag: "node"; left: FTCQueue<I, A, unknown>; right: FTCQueue<I, unknown, B> };

type Freer<I, A> =
  | { tag: "pure"; value: A }
  | { tag: "impure"; op: I; queue: FTCQueue<I, unknown, A> }
  | { tag: "suspend"; thunk: () => Freer<I, A> };
Не копируй это как финальный код вслепую. Это форма, которую надо адаптировать под текущие src/freer/types.ts, freerMk и Symbol.iterator. Домашняя работа здесь — провести миграцию, не сломав внешний смысл.

Что происходит с smart constructors

Раньше конструктор инструкции делал cont: freerPure. После перехода на очередь он должен создавать очередь из одного продолжения: 

// было: cont: freerPure
// стало по смыслу: queue: singleton(freerPure)

Для writeLine, sleep и других void-эффектов continuation всё равно принимает ответ инструкции. Просто этот ответ не используется.

Takeaway. Impure теперь хранит “текущую инструкцию + весь хвост программы”, а не “текущую инструкцию + одну функцию”.

FTCQueue: минимальная структура

FTCQueue расшифровывают как Fast Type-aligned Catenable Queue. Для учебной версии нужны три операции:

Операция Смысл Стоимость
singleton(f) очередь из одного continuation O(1)
append(q1, q2) или snoc(q, f) добавить continuation без обхода старой программы O(1)
viewL(q) вынуть первый continuation и остаток очереди амортизированная O(1)

Почему не обычный массив

Для учебного JS-кода массив кажется проще, но он маскирует идею. Если делать push, добавление дешёвое, но shift в JS-массиве обычно двигает элементы. Если хранить индекс головы, можно получить нормальную практическую структуру, но потеряется связь с paper-а. FTCQueue показывает именно algebraic решение: очередь как дерево композиций.

Учебный ViewL

type ViewL<I, A, B> =
  | { tag: "one"; f: Cont<I, A, B> }
  | { tag: "cons"; head: Cont<I, A, unknown>; tail: FTCQueue<I, unknown, B> };

viewL должен вернуть первый continuation. Если очередь — один leaf, остатка нет. Если очередь — node(left, right), нужно найти голову в left, а остаток склеить обратно с right.

Практический критерий. Если твой bind вызывает себя рекурсивно по старому m.cont(resp), ты ещё не решил E12.2. После миграции bind для impure должен только добавить функцию в очередь.

Takeaway. Очередь нужна не ради красоты типов, а чтобы разнести “добавить продолжение” и “исполнить следующее продолжение”.

План миграции для текущего проекта

Делай миграцию маленькими шагами. Не переписывай smart constructors и interpreter одновременно.

Шаг 1. Добавить модуль очереди

Создай отдельный модуль, например src/freer/ftc-queue.ts. Он не должен импортировать конкретные инструкции ReadLine, Fetch, Random. Максимум — общий тип Freer или функцию-continuation. 

// форма API, не готовое решение:
singleton(f)
append(left, right)
snoc(queue, f)
viewL(queue)

Шаг 2. Поменять FreerImpure

В src/freer/types.ts заменить поле cont на поле очереди. Лучше не называть поле cont, чтобы TypeScript и глаза не смешивали старую и новую модель.

Шаг 3. Обновить smart constructors

Каждый конструктор impure должен создавать очередь из одного continuation. Например, readLine отвечает строкой, значит первый continuation превращает эту строку в pure(string). writeLine отвечает void, значит continuation возвращает pure(undefined).

Шаг 4. Переписать freerBind

Старый recursive case: 

case "impure":
  return impure(op, (resp) => bind(cont(resp), f));

Новый смысл: 

case "impure":
  return impure(op, snoc(queue, f));

Для pure всё остаётся: f(value). Для suspend можно сохранить ленивую границу: suspend(() => bind(thunk(), f)). Это не главная цель E12.2, но сохраняет свежий generator.

Шаг 5. Переписать interpreter

После выполнения инструкции interpreter получает resp. Раньше он делал current = current.cont(resp). Теперь он должен вынуть голову очереди и применить её к ответу. 

// форма шага:
const next = applyQueue(current.queue, resp);
current = next;

Если head(resp) вернул pure, нужно либо продолжить хвост очереди с этим значением, либо завершить, если хвоста нет. Если вернул impure, хвост старой очереди надо присоединить к очереди нового impure. Это и есть ключевой момент “reflection without remorse”.

Шаг 6. Проверить E11

После миграции должны остаться зелёными: 

pnpm exec vitest run test-vasya/exercises-part11.test.ts
pnpm exec vitest run test-vasya/monad-laws.test.ts
pnpm exec vitest run test-vasya/integration.test.ts

Takeaway. E12.2 считается сделанным не когда код “похож на очередь”, а когда старое поведение Freer сохранено, а left-associated benchmark перестал расти квадратично.

Бенчмарк для подтверждения O(N)

Бенчмарк должен измерять построение left-associated цепочки. Если включить в замер реальные эффекты, шум interpreter-а может скрыть проблему. 

function buildLeftAssociated(n: number) {
  let prog = freerPure(0);
  for (let i = 0; i < n; i++) {
    prog = freerBind(prog, (x) => freerPure(x + 1));
  }
  return prog;
}

Для наивного Freer рост должен быть близок к квадратичному или упереться в stack overflow. Для FTCQueue построение должно стать примерно линейным. Не жди идеального графика: V8, GC и JIT будут шуметь. Важна форма роста между 1k, 10k и 100k.

Vitest-форма проверки

import { describe, expect, it } from "vitest";

describe("E12.2 benchmark shape", () => {
  it("builds a left-associated chain without stack overflow", () => {
    const p = buildLeftAssociated(100_000);
    expect(p).toBeDefined();
  });
});

Отдельный benchmark-скрипт может печатать таблицу. Unit-тест не должен быть хрупким по миллисекундам: CI и локальная машина дадут разные числа.

N Наивный Freer Freer + FTCQueue
1k обычно проходит проходит
10k заметно медленнее или stack pressure проходит с умеренным ростом
100k часто stack overflow / очень долго строится без рекурсивного обхода цепочки

Takeaway. Бенчмарк должен ловить не “быстро/медленно”, а смену класса сложности.

Типичные ошибки

  • Оставить рекурсивный bind в impure. Тогда очередь есть в типах, но не в алгоритме.
  • Использовать Array.shift() как очередь. Это может вернуть линейную стоимость на извлечение головы.
  • Забыть склеить хвосты. Если continuation вернул новый impure, старый хвост очереди нельзя потерять.
  • Сломать suspend. doFreer всё ещё должен создавать свежий generator на каждый run.
  • Бенчмаркать только pure-цепочку. Она может схлопываться слишком рано. Проверь и pure-chain, и цепочку через хотя бы одну suspended/impure границу.

Research brief

Scope

Узкая тема: как заменить continuation в учебном Freer на FTCQueue и что должен делать interpreter. Вне scope: полноценные open unions, Effect-TS runtime, конкуренция fibers, typed error channel.

Prerequisites

  • Текущая часть 11: Freer<I, A>, impure, op, cont.
  • Monad laws: особенно associativity.
  • Базовая амортизированная сложность структур данных.
  • TypeScript generics и готовность локально использовать unknown внутри unsafe boundary.

Рекомендуемый порядок чтения

  1. Перечитай docs/freer-part11-explainer.html: восстанови текущую модель Freer.
  2. Прочитай abstract и Sections 1-2 paper-а “Reflection without Remorse”: там объясняется left-associated bind как общий structural problem.
  3. Посмотри Haddock freer-simple: Eff хранит Union effs b и Arrs effs b a, где Arrs — FTCQueue.
  4. Вернись в проект и нарисуй переход: cont → queue, current.cont(resp) → applyQueue(queue, resp).
