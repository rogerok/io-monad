Freer: как перестать переписывать bind при каждом новом эффекте

В монолитном IO каждый эффект живёт внутри общего типа, а bind знает все варианты. Freer меняет центр тяжести: bind работает с формой Pure | Impure, а конкретные инструкции остаются данными для интерпретатора.

Этот файл не даёт готовую реализацию задания. Он показывает форму решения, трассировку вычислений и вопросы, по которым можно проверить свою архитектуру.

2. Расщепление: инструкция больше не содержит next

Монолитная форма

У каждой инструкции внутри лежит знание, что делать дальше. Поэтому bind обязан залезать в каждый вариант. 

type ReadLine<A> = {
  tag: "readLine";
  next: (s: string) => IO<A>;
};

type WriteLine<A> = {
  tag: "writeLine";
  text: string;
  next: IO<A>;
};

Freer-форма

Инструкция описывает только команду. Продолжение вынесено в общий узел impure. 

type ReadLine = {
  tag: "readLine";
  _resp: string;
};

type WriteLine = {
  tag: "writeLine";
  text: string;
  _resp: void;
};

type Freer<I, A> =
  | { tag: "pure"; value: A }
  | { tag: "impure"; op: I; cont: (resp: any) => Freer<I, A> };

3. Почему bind больше не меняется

Один алгоритм

const bind = <I, A, B>(
  m: Freer<I, A>,
  f: (a: A) => Freer<I, B>,
): Freer<I, B> => {
  if (m.tag === "pure") return f(m.value);

  return {
    tag: "impure",
    op: m.op,
    cont: (resp) => bind(m.cont(resp), f),
  };
};

Чтение по строкам

1
pure Значение уже есть, поэтому можно просто применить функцию продолжения f.
2
impure Эффект ещё не исполнен. Его нельзя раскрыть внутри bind, потому что bind не интерпретатор.
3
Композиция Старый cont получит ответ эффекта, а затем результат будет связан с новым f.

Ключевой вопрос для самопроверки: если добавить random, какая строка в этом bind должна измениться? Правильный ответ: ни одна.

Случай A: bind(pure(2), f)

Здесь эффекта нет. Значение 2 уже лежит в руках у bind. Поэтому bind не строит новый impure-узел, а сразу вызывает f(2). 

const f = (n: number) => writeLine(String(n + 1));

bind(pure(2), f)

// подстановка:
if (m.tag === "pure") return f(m.value)

// результат построения:
writeLine("3")

Важно: writeLine("3") ещё не напечатал строку. Это только новое дерево программы. Печать случится позже, когда интерпретатор дойдёт до этого узла.

Случай B: bind(readLine, f)

Здесь значения ещё нет. Есть только инструкция readLine, которая когда-то вернёт строку. Поэтому bind не может вызвать f(name) прямо сейчас: name появится только во время исполнения. 

const f = (name: string) => writeLine("Hi, " + name);

bind(readLine, f)

// результат построения:
{
  tag: "impure",
  op: { tag: "readLine" },
  cont: (resp) => bind(oldCont(resp), f)
}

f не потерялся. Он переехал внутрь нового cont и будет вызван только после ответа от readLine.

Две разные фазы: построение и исполнение

Фаза 1. bind строит программу

Он смотрит только на форму узла: pure или impure. Он не читает ввод, не пишет в консоль, не делает fetch и не генерирует random.

Фаза 2. runIO исполняет программу

Интерпретатор берёт op, выполняет эффект в world, получает resp и вызывает current.cont(resp). 

// Построение:
const program = bind(readLine, (name) => writeLine("Hi, " + name));

// Исполнение:
current = program                         // impure readLine
resp = await world.readLine()             // например "Alice"
current = current.cont(resp)              // теперь построится writeLine("Hi, Alice")
await world.writeLine("Hi, Alice")
current = current.cont(undefined)         // pure(undefined)

5. Что означает _resp

Фантомный ответ

Поле _resp говорит TypeScript: “если интерпретатор исполнит эту инструкцию, ответ будет такого типа”. В рантайме это поле обычно не нужно создавать. 

type ReadLine = {
  tag: "readLine";
  _resp: string;
};

type Random = {
  tag: "random";
  _resp: number;
};

Зачем это smart constructor

Конструктор знает конкретную инструкцию и может вернуть Freer<Instr, string> для readLine или Freer<Instr, number> для random. 

// Подсказка формы, не готовый код проекта
readLine : Freer<Instr, string>
writeLine(text) : Freer<Instr, void>
random : Freer<Instr, number>

Если в генераторе const n = yield* random, тип n должен стать number. Это и есть практическая польза _resp.

6. runWithLogging как декоратор интерпретатора

Не новый Freer

Логирование не должно менять Freer и bind. Это другая политика исполнения: перед каждой инструкцией записать op.tag, затем выполнить ту же инструкцию обычным способом.

Ядро pure impure

Не знает про логирование.

Интерпретатор switch op.tag log

Решает, что делать с инструкциями.

Ментальная трасса

program:
  writeLine("hi")
  readLine
  writeLine("echo " + name)

runWithLogging:
  log("writeLine")
  world.writeLine("hi")
  log("readLine")
  name = world.readLine()
  log("writeLine")
  world.writeLine("echo " + name)

В тестовом мире результат программы тот же, но рядом появляется отдельный массив логов.

7. Дорожная карта к E11.1, E11.2, E11.3

E11.1

Переезд на Freer

Сначала отдели универсальный тип Freer<I, A> и универсальный bind. Затем опиши конкретный Instr для IO-инструкций.

Самопроверка: бизнес-программа должна выглядеть почти так же, как раньше. Если она переписывается радикально, граница выбрана неудачно.

E11.2

Random

Добавление эффекта должно затронуть набор инструкций, smart constructor и интерпретатор. Универсальные Freer и bind остаются закрытыми.

Самопроверка: можно удалить ветку random из интерпретатора и получить ошибку только на уровне исполнения/исчерпывающего switch, но не в bind.

E11.3

runWithLogging

Думай о нём как о втором интерпретаторе или декораторе мира. Он видит каждую инструкцию, логирует tag, потом делегирует реальную работу.

Самопроверка: output программы не должен ломаться из-за логирования. Логи являются дополнительным наблюдением, а не заменой исполнения.
