Freer: как программа движется под капотом

Freer IO хранит программу как данные. Узел impure говорит: “выполни запрос op, получи ответ resp, затем передай его в cont, чтобы построить следующий узел”. Поэтому bind больше не обязан знать про readLine, writeLine, fetch или будущий random.

Это учебный материал, а не готовое решение задания. Он объясняет поток выполнения, форму данных и точки расширения, чтобы ты мог проверить собственную реализацию.

1. Карта выполнения

Самая частая путаница: смешиваются построение программы и её исполнение. В Freer они разделены. Пользовательский код собирает значение-программу, а runIO позже интерпретирует это значение через world.

T0
Ты описываешь программу. Вызовы smart constructor вроде readLine или writeLine("hi") создают узлы. Побочных эффектов ещё нет.
T1
bind сцепляет куски. Если значение уже pure, он применяет функцию сразу. Если перед ним impure, он не исполняет эффект, а дописывает новую функцию в continuation.
T2
runIO становится циклом исполнения. Он держит current, смотрит на current.tag, выполняет current.op в world и получает resp.
T3
cont(resp) возвращает следующий узел. Интерпретатор не знает бизнес-логики “что после имени”. Она живёт в continuation, которое было создано при построении программы.
while current is impure:
  resp = interpret(current.op, world)
  current = current.cont(resp)

Если держать в голове только одну фразу, держи эту: op — запрос наружу, resp — ответ наружного мира, cont — продолжение программы после ответа.

2. Какие данные лежат в Freer

Старый монолитный IO

В ранней модели каждый вариант эффекта сам хранил свой способ продолжить программу. Из-за этого bind должен был знать форму каждого варианта. 

type ReadLine<A> = {
  tag: "readLine";
  next: (s: string) => IO<A>;
};

type WriteLine<A> = {
  tag: "writeLine";
  text: string;
  next: IO<A>;
};

Freer-форма

Инструкция стала маленьким запросом. Общая оболочка impure хранит сам запрос и единое место для continuation. 

type Freer<I, A> =
  | { tag: "pure"; value: A }
  | { tag: "impure"; op: I; cont: (resp: unknown) => Freer<I, A> }
  | { tag: "suspend"; thunk: () => Freer<I, A> };
Часть Что означает Кто её использует Типичная ошибка понимания
pure(value) Программа уже завершилась обычным значением. bind и runIO. Думать, что pure запускает вычисление. Он только хранит результат.
op Описание одного запроса: прочитать строку, напечатать текст, сходить в сеть. Только конкретный interpreter. Класть в op бизнес-логику “что делать потом”.
cont Функция, которая принимает ответ эффекта и возвращает следующий Freer-узел. bind строит, runIO вызывает. Думать, что это JS call stack. Это значение в дереве программы.
suspend Отложенное построение следующего куска программы. runIO, когда доходит до suspend-узла. Создавать generator заранее и случайно делать программу одноразовой.
_resp Тип ответа конкретной инструкции: readLine -> string, random -> number. TypeScript и smart constructors. Искать это поле в runtime. Обычно это фантомная подсказка типам.

3. Что делает bind на самом деле

bind(m, f) означает: “возьми программу m, дождись её обычного результата A, затем продолжи через f: A => Freer<I, B>”. Но если m ещё ждёт эффект, результата A пока нет. Тогда bind не может вызвать f прямо сейчас.

Псевдокод ядра

bind(m, f):
  if m is pure(value):
    return f(value)

  if m is suspend(thunk):
    return suspend(() => bind(thunk(), f))

  if m is impure(op, oldCont):
    return impure(
      op,
      resp => bind(oldCont(resp), f)
    )

Смысл трёх веток

pure: результат уже есть f вызывается сразу, потому что нечего ждать у внешнего мира.
suspend: результат появится после форсирования thunk bind оборачивает thunk, чтобы сохранить ленивость и не запускать generator раньше времени.
impure: результат появится после op f прячется внутрь нового cont. Сам op переносится без изменений.

Почему это освобождает от switch по эффектам

В ветке impure bind не спрашивает, какой это эффект. Для него readLine, writeLine, fetch и будущий random имеют одну форму: есть op и есть cont. Это и есть главный выигрыш Freer.

Что сохраняется как есть

op. Если это был { tag: "readLine" }, он таким и остаётся.

Что меняется

cont. Старое продолжение соединяется с новой функцией f.

Проверка понимания: если при добавлении random приходится менять универсальный bind, значит инструкция и монадическая механика снова сцепились.

4. Что делает runIO

runIO — это interpreter. Он единственный имеет право превращать op в настоящий эффект. Внутри удобно думать о четырёх регистрах: current, op, resp, locals.

current

Узел, на котором стоит interpreter прямо сейчас: pure, impure или suspend.

op и resp

op уходит во внешний мир. resp возвращается из внешнего мира и передаётся в cont.

locals

Локальные переменные программы не лежат в op. Они живут внутри замыканий continuation и “появляются”, когда continuation вызывается.

Скелет interpreter без деталей задания

current = program

loop:
  if current is pure(value):
    finish with value

  if current is suspend(thunk):
    current = thunk()
    continue loop

  if current is impure(op, cont):
    resp = execute op using world
    current = cont(resp)
    continue loop

Здесь нет готового switch по всем инструкциям. Важно устройство потока: interpreter исполняет один запрос, получает один ответ и только потом строит следующий узел.

5. Как freerDo связан с generator и yield*

yield* не исполняет эффект. Он отдаёт наружу Freer-узел, а freerDo превращает шаги generator в цепочку bind. В этой кодовой базе есть важная деталь: generator должен создаваться внутри suspend, чтобы программа могла быть отложенной и не расходовать одноразовый iterator до запуска.

Что видит автор программы

freerDo(function* () {
  yield* writeLine("What is your name?");
  const name = yield* readLine;
  yield* writeLine("Hello, " + name);
});

Синтаксис выглядит императивно, но это построение Freer-значения. Строка const name = ... получит значение только после того, как interpreter выполнит readLine и вернёт ответ в generator.

Что происходит под капотом

freerDo returns:
  suspend(() => {
    gen = genFn()
    walk(undefined)
  })

walk(resp):
  result = gen.next(resp)
  if result.done: pure(result.value)
  else: bind(result.value, nextResp => suspend(() => walk(nextResp)))

walk делает один шаг generator. Если generator отдал Freer-инструкцию, она связывается с continuation, которое позже вернёт ответ обратно в gen.next(answer).

Важная граница: freerDo управляет generator-ом, но не выполняет readLine или writeLine. Эффекты всё ещё выполняет только interpreter.

7. Добавление Random и logging

новый эффект

Random

Добавление эффекта должно расширить словарь инструкций и concrete interpreter. Но универсальный Freer и bind не меняются: для них это всё ещё impure(op, cont). 

Random instruction:
  op.tag = "random"
  response type = number

Interpreter:
  resp = world.random()
  current = cont(resp)
новая политика исполнения

runIOWithLogging

Logging — не новый Freer. Это другой interpreter или декоратор interpreter-а: перед выполнением инструкции он записывает наблюдение, затем делегирует обычное исполнение. 

for each impure node:
  log(op.tag and useful fields)
  resp = interpret(op, world)
  current = cont(resp)
Изменение Где оно должно быть Где его не должно быть
Новый random В union инструкций, smart constructor, world, interpreter. В универсальном bind.
Логирование В interpreter/decorator, который видит каждый op. В smart constructors и в Freer-ядре.
Новая бизнес-логика В программе: в функциях, переданных в bind, или внутри generator. В runIO. Interpreter не должен знать сценарий пользователя.

8. Самопроверка перед кодом

Мини-вопрос 1

Почему bind(readLine, f) не может сразу вызвать f?

Потому что аргумент для f появится только после исполнения readLine в interpreter.

Мини-вопрос 2

Где хранится “после имени напечатать приветствие”?

В continuation, которое получит имя как resp и построит следующий Freer-узел.

Мини-вопрос 3

Почему logging не должен менять результат программы?

Потому что logging добавляет наблюдение за op, но не заменяет делегирование в обычный interpreter.

9. Источники и как их читать

Для этой части не нужно закрывать всю теорию extensible effects. Достаточно взять из источников три идеи: computation as data, explicit continuation и interpreter as handler.

интуиция

Free Monoids and Free Monads, Free of Category Theory

Полезен как вход: free monad можно воспринимать как структуру, которая сохраняет описание вычисления и позволяет позже интерпретировать его разными способами.

основной paper

Freer Monads, More Extensible Effects

Главный источник: continuation выносится из request signature в общий Impure, из-за чего исчезает требование Functor и упрощается добавление эффектов.

ориентация

Papers We Love: Freer Monads

Короткая карточка paper-а: полезна, чтобы увидеть связь Freer, extensible effects, handlers и нескольких interpreter-ов.
