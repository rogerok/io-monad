Freer: как `src/freer` отделяет программу от интерпретатора

В монолитной IO-монаде каждый эффект был вариантом общего `IO` и `bind` знал про все теги. Freer меняет форму данных: инструкция хранит только запрос, а продолжение выносится в универсальное поле `cont`. Поэтому `freerBind` работает одинаково для `readLine`, `writeLine`, `fetch`, `random`, `sleep` и будущих эффектов.

Контекст. Эта страница объясняет текущую реализацию из `src/freer/*`: `types.ts`, `constructors.ts`, `freer-mk.ts`, `bind.ts`, `freer-do.ts`, `freer-run.ts` и `program.ts`. Она не заменяет домашнее задание и не требует менять `index.ts`: цель - увидеть механику уже собранного Freer encoding.

1. Карта идеи

Программа

`FreerIO<A>` - значение. Оно может быть `pure`, `suspend` или `impure`. В `impure` лежит один запрос `op` и функция `cont`, которая знает, как продолжить программу после ответа на запрос.

данные yield*

Композиция

`freerBind(m, f)` не исполняет эффекты. Он либо применяет `f` к готовому `pure`, либо проталкивает `f` внутрь будущего продолжения. Ветка `impure` не смотрит на `op.tag`.

bind cont

Интерпретация

`freerRun` идёт по программе циклом. Только здесь `op.tag` превращается в реальные действия из `FreerWorld`: чтение, печать, HTTP, random, sleep.

effects World
smart constructor freerWriteLine("hi")

создаёт `impure` с `op` и начальным `cont`

→
bind / do freerBind(node, f)

пересобирает continuation, не трогая мир

→
interpreter freerRun(program, world)

выполняет `op` и вызывает `cont(resp)`

Вывод. Freer делает явной границу: ядро знает только `pure`, `suspend`, `impure`; конкретные инструкции знает только interpreter.

2. Форма данных в `src/freer/types.ts`

В монолитном `IO` у каждой инструкции был свой `next`: у `ReadLine` это `(s: string) => IO<A>`, у `WriteLine` это `IO<A>`, у `Fetch` это `(body: string) => IO<A>`. Freer нормализует форму: ответ инструкции описывается фантомным `_resp`, а продолжение всегда лежит рядом с `op`.

Инструкция: только запрос

export type FreerReadLine = {
  _resp: string;
  tag: "readLine";
};

export type FreerWriteLine = {
  _resp: void;
  tag: "writeLine";
  text: string;
};

export type FreerRandom = {
  _resp: number;
  tag: "random";
};

`_resp` не используется в runtime. Это метка для TypeScript: “если interpreter выполнит такую инструкцию, ответ будет этого типа”.

Freer-узел: запрос + continuation

export type FreerImpure<I extends Instr<any>, A> = {
  tag: "impure";
  op: I;
  cont: (resp?: any) => Freer<I, A>;
};

export type Freer<I extends Instr<any>, A> = {
  [Symbol.iterator](): FreerGen<A>;
} & RawFreer<I, A>;

`A` - финальный результат всей программы. `op` не должен знать этот `A`. Его знает `cont`.

Поле Где живёт Что означает Почему важно
op FreerImpure Один запрос к внешнему миру: `readLine`, `fetch`, `sleep`. Инструкции больше не содержат продолжение.
cont FreerImpure Функция “ответ эффекта -> следующая программа”. `bind` может композиционно менять только continuation.
_resp Instr Фантомный тип ответа инструкции. Типизирует smart constructors без хранения runtime-поля.
suspend RawFreer Отложенная сборка поддерева. Даёт свежий generator при каждом `freerRun`.
Вывод. Freer не делает эффекты “магическими”. Он меняет представление AST так, чтобы `bind` не зависел от списка инструкций.

3. Smart constructors: минимальный `impure`

Каждый constructor создаёт один узел `impure`. Его `cont` сначала просто возвращает результат в `freerPure`. Позже `freerBind` заменит это продолжение на более длинное.

`freerReadLine`

export const freerReadLine: FreerIO<string> = freerMk({
  tag: "impure",
  op: { tag: "readLine" } as FreerReadLine,
  cont: freerPure,
});

Ответ `readLine` - строка. Поэтому начальное продолжение равно `freerPure`: строка становится итоговым значением этой маленькой программы.

`freerWriteLine`

export const freerWriteLine = (text: string): FreerIO<void> =>
  freerMk({
    tag: "impure",
    op: { tag: "writeLine", text } as FreerWriteLine,
    cont: () => freerPure(undefined),
  });

`writeLine` не возвращает полезное значение, поэтому `cont` игнорирует ответ interpreter-а и завершает узел `undefined`.

Типовая ловушка. `freerReadLine` здесь значение, а не функция. Это важно для той же причины, что и в обычном `IO`: само значение уже является отложенным описанием программы.

4. Почему `freerBind` не знает про `readLine` и `fetch`

В Freer `bind` работает с тремя формами: `pure`, `suspend`, `impure`. Конкретный эффект спрятан внутри `op`, но `bind` не нужно его понимать. Ему достаточно знать, где лежит продолжение. 

export const freerBind = <I extends Instr<any>, A, B>(
  m: Freer<I, A>,
  f: (a: A) => Freer<I, B>,
): Freer<I, B> => {
  switch (m.tag) {
    case "pure":
      return f(m.value);

    case "suspend":
      return freerMk({
        tag: "suspend",
        thunk: () => freerBind(m.thunk(), f),
      });

    case "impure":
      return freerMk({
        tag: "impure",
        op: m.op,
        cont: (resp) => freerBind(m.cont(resp), f),
      });
  }
};

`pure`

Значение уже готово, значит можно сразу вызвать `f(value)`. Это обычное правило `bind(Pure(a), f) = f(a)`.

`suspend`

Поддерево ещё не создано. `bind` сохраняет ленивую границу и добавляет `f` внутрь будущего `thunk`.

`impure`

Запрос `op` остаётся прежним. Меняется только `cont`: после ответа сначала строится старый хвост, затем к нему приклеивается `f`.

Вывод. Новый эффект не требует правки `freerBind`, потому что `bind` больше не делает switch по `op.tag`.

5. `freerDo`, `yield*` и `freerMk`

`freerDo` даёт линейный синтаксис поверх тех же `impure`-узлов. Важная деталь: generator создаётся внутри `freerSuspend`, поэтому один и тот же `program` можно запускать несколько раз. Без `suspend` generator был бы одноразовым состоянием, зашитым в программу.

Как `yield* freerReadLine` становится typed

export const freerMk = <I extends Instr<any>, A>(
  freer: RawFreer<I, A>,
): Freer<I, A> => {
  Object.defineProperty(freer, Symbol.iterator, {
    enumerable: false,
    value: function* () {
      return (yield new YieldWrap(this)) as A;
    },
  });

  return freer as Freer<I, A>;
};

`yield*` вызывает `Symbol.iterator`. Итератор отдаёт наружу `YieldWrap(this)`, а возвращаемый тип итератора становится типом выражения `yield*`.

Как generator превращается в Freer

export const freerDo = <A>(genFn: () => FreerGen<A>): FreerIO<A> =>
  freerSuspend(() => {
    const gen = genFn();

    const walk = (v: unknown): FreerIO<A> => {
      const result = gen.next(v);

      if (result.done) {
        return freerPure(result.value);
      }

      return freerBind(
        result.value.value,
        (v) => freerSuspend(() => walk(v)),
      );
    };

    return walk(undefined);
  });

`result.value` - это `YieldWrap`. Его `.value` - сам Freer-узел, который нужно связать с продолжением `walk`.

Вывод. `yield*` не выполняет эффект. Он отдаёт Freer-узел в `freerDo`; выполнение начнётся только в `freerRun`.

6. `freerRun`: единственный switch по инструкциям

Interpreter получает `FreerIO<A>` и `FreerWorld`. Пока текущий узел не `pure`, он либо раскрывает `suspend`, либо берёт `current.op`, выполняет соответствующий метод мира и передаёт результат в `current.cont`. 

export const freerRun = async <A>(
  freer: FreerIO<A>,
  world: FreerWorld,
): Promise<A> => {
  let current: FreerIO<A> = freer;

  while (true) {
    if (current.tag === "pure") return current.value;
    if (current.tag === "suspend") {
      current = current.thunk();
      continue;
    }

    const op = current.op;

    switch (op.tag) {
      case "readLine":
        current = current.cont(await world.readLine());
        break;
      case "writeLine":
        await world.writeLine(op.text);
        current = current.cont();
        break;
      case "fetch":
        current = current.cont(await world.fetch(op.url, op.options));
        break;
      case "random":
        current = current.cont(world.random());
        break;
      case "sleep":
        await world.sleep(op.ms);
        current = current.cont();
        break;
      case "fail":
        throw op.error;
    }
  }
};
op.tag World-вызов Что передаётся в `cont`
readLine await world.readLine() string
writeLine await world.writeLine(op.text) undefined
fetch await world.fetch(op.url, op.options) string
random world.random() number
sleep await world.sleep(op.ms) undefined
Вывод. Все side effects локализованы в `freerRun` и `FreerWorld`. Сам `program` можно запускать в Node, браузере, тестах или через logging-world.

8. Почему добавление `Random` не трогает `bind`

В текущем `src/freer` уже есть эффект `random`. Его добавление требует три локальные вещи: тип инструкции, smart constructor, ветку interpreter-а. `freerBind` и `freerDo` остаются прежними, потому что для них `random` - обычный `impure`.

1. Тип инструкции

export type FreerRandom = {
  _resp: number;
  tag: "random";
};

2. Constructor

export const freerRandom: FreerIO<number> = freerMk({
  tag: "impure",
  op: { tag: "random" } as FreerRandom,
  cont: freerPure,
});

3. Interpreter

case "random": {
  current = current.cont(world.random());
  break;
}

Это и есть практический смысл “открыт для расширения”: монадическое ядро не переписывается при появлении новой команды DSL. В TypeScript union `Instr` всё ещё закрыт на уровне файла, но архитектурно зависимость сдвинута из `bind` в interpreter.

Граница реализации. Эта версия Freer ещё не использует `FTCQueue`. Поэтому длинные левоассоциированные цепочки могут сохранять асимптотические проблемы наивного Free/Freer. Историческая статья “Reflection without Remorse” решает это через type-aligned sequence.
